# Ngspice FP32 混合精度改造：FP64 保留策略分析

## 概述

本文档系统性地记录了 Ngspice 在 `--enable-single-precision` 构建中，**哪些模块仍然使用 FP64**，以及保留 FP64 的理由。这是分层混合精度策略的核心设计文档。

---

## 总体策略

```
FP32 = 默认工作精度（存储、带宽）
FP64 = 保留用于三类场景：
       1. 无法改：第三方库硬编码 double
       2. 不必改：整数/索引/控制逻辑
       3. 改不动：裸数学函数调用（~500+文件，非阻塞性）
```

---

## 第一类：无法改为 FP32（结构性障碍）

### 1.1 KLU 稀疏求解器

| 属性 | 说明 |
|------|------|
| 位置 | `src/maths/KLU/` (18 个源文件) |
| 现状 | 已通过 `--disable-klu` 禁用，回退到 Sparse 1.3 |
| 硬编码位置 | `klu_version.h:263-264`: `typedef double Unit; #define Entry double` |
| 原因 | SuiteSparse 第三方库，无类型抽象层，无 float 变体 |
| 无法改的理由 | 1) 需改动 KLU 所有内部算法的精度假设（pivot 容差、缩放等）；2) KLU 有 4 个名称修饰变体（klu_/klu_z_/klu_l_/klu_zl_），只改变整数宽度和实/复数，不涉及标量类型；3) 与上游 SuiteSparse 分叉会产生持续维护负担 |
| 影响 | 大规模病态矩阵（条件数 > 1e6）的回退路径不可用。对 AnalogSizing 的 ~30×30 运放矩阵无影响 |

### 1.2 KLU 包装层 (klusmp.c)

| 属性 | 说明 |
|------|------|
| 位置 | `src/maths/KLU/klusmp.c` |
| 硬编码位置 | 43 处 `(double *) malloc(... sizeof(double))` |
| 原因 | 该包装层将 SPICE_REAL 矩阵元素通过指针重定向映射到 KLU 的 double* CSC 数组。FP32 下，器件写入 4 字节 float 到 8 字节 double 槽，高 4 字节为垃圾值——这是严重的类型不匹配 bug |
| 影响范围 | 55 个器件绑定文件 (`*/*bindCSC.c`) |
| 无法改的理由 | 修复需要在每个 Newton 迭代中将整个矩阵从 float 拓宽为 double 再调用 KLU，求解后再从 double 缩窄为 float——这既增加了运行时开销，又抵消了 FP32 的内存优势 |

### 1.3 XSPICE 代码模型

| 属性 | 说明 |
|------|------|
| 位置 | `src/xspice/` |
| 现状 | 已通过 `--disable-xspice` 禁用 |
| 硬编码位置 | `cm.c:115,262,355`: `doubles_needed = bytes / sizeof(double) + 1` |
| 原因 | XSPICE 状态管理使用 `sizeof(double)` 计算状态数组索引。FP32 下 `sizeof(SPICE_REAL)=4` ≠ `sizeof(double)=8`，导致索引偏差 2 倍→越界访问。此外 `delay/cfunc.c` 中的 `CKTcircuitmin` 结构体所有字段硬编码为 `double*` 而非 `SPICE_REAL*` |
| 影响范围 | `cm.c` + 12+ 代码模型文件 (`*icm/analog/*/cfunc.c`) |
| 无法改的理由 | 改动波及面大（每个代码模型的状态管理都需要修改），且 AnalogSizing 不需要 XSPICE（纯电路仿真，无事件驱动） |

## 第二类：不必改为 FP32（无收益）

### 2.1 电路拓扑与连接关系

| 属性 | 说明 |
|------|------|
| 位置 | 所有器件的 setup/bind 函数 |
| 原因 | 节点索引、非零元位置、器件连接关系——全是 `int`，与浮点精度无关 |
| 不改的理由 | 零风险，零收益 |

### 2.2 网表解析器

| 属性 | 说明 |
|------|------|
| 位置 | `src/spicelib/parser/` |
| 原因 | 字符串→数值转换。`IFvalue.rValue` 已定义为 `SPICE_REAL`，解析出的数值以 FP32 精度存储就是正确的 |
| 不改的理由 | 解析精度 ≤ FP32 表示精度，无额外损失 |

### 2.3 AMD / BTF / COLAMD

| 属性 | 说明 |
|------|------|
| 位置 | `src/maths/KLU/amd_*.c`, `btf_*.c`, `colamd.c` |
| 原因 | 这三个库是 KLU 的依赖，但它们是**纯符号算法**（图排序/二分图匹配/强连通分量分解），从不接触矩阵数值 |
| 不改的理由 | 它们 API 中的 `double Control[]` / `double Info[]` 数组仅传递控制参数和统计信息，无精度敏感性 |

### 2.4 前端/命令行/绘图

| 属性 | 说明 |
|------|------|
| 位置 | `src/frontend/` |
| 原因 | UI 逻辑、输出格式化、绘图数据——显示精度远低于 FP32 |
| 不改的理由 | 非数值计算，无精度/性能需求 |

## 第三类：改不动（非阻塞性）

### 3.1 裸 `double` 数学函数调用

这是代码库中**规模最大的未转换区域**，但**在当前 AnalogSizing 场景下无实际影响**。

| 属性 | 说明 |
|------|------|
| 位置 | 每个器件模型文件和每个分析文件（~500+ 文件） |
| 规模 | `exp()` ~300次, `log()` ~200次, `sqrt()` ~200次, `pow()` ~100次, `fabs()` ~500次, `sin/cos` ~60次 |
| 原因 | `typedefs.h` 中定义了 `SPICE_EXP`/`SPICE_LOG`/`SPICE_SQRT` 等宏，在 FP32 下映射到 `expf`/`logf`/`sqrtf`。但**整个仿真代码库中没有一个文件使用这些宏**——所有代码直接调用裸 `double` 数学函数 |
| 编译器行为 | 隐式 float→double 提升 → 调用 double 数学库函数 → 结果截断回 float |
| 不改的理由 | 1) 改动规模过大（~500+ 文件），风险收益比不合理；2) 编译器隐式转换已保证正确性（计算结果在 double 下进行后再截断，精度反而更高）；3) 对 AnalogSizing 的小矩阵场景，数学函数调用不构成性能瓶颈；4) **真正的性能瓶颈在内存带宽和 SIMD 吞吐**，而非数学函数的调用开销 |

### 3.2 complex.h 复数宏

| 属性 | 说明 |
|------|------|
| 位置 | `src/include/ngspice/complex.h` (73-555 行) |
| 硬编码 | `cmag`, `cph`, `cdiv`, `C_ABS`, `C_SQRT`, `C_NORM` 等宏直接调用裸 `fabs()`, `sqrt()`, `hypot()`, `atan2()` |
| 不改的理由 | DC 工作点分析不涉及复数运算。AC/噪声分析采用 FP64 基线。当需要 FP32 AC 分析时再处理 |

### 3.3 FP64 专用常量

下表列出了所有硬编码 FP64 常量的位置、值和问题。**当前 AnalogSizing 场景下这些常量的影响极小**（器件偏置在强反型区，避开了亚阈值数值陷阱）。

| 常量 | 位置 | 值 | FP32 下的问题 | 影响评估 |
|------|------|-----|-------------|---------|
| `DBL_EPSILON` | `dctrcurv.c:217,230,243,255,460` | 2.22e-16 | 步长阈值远小于 FP32 可表示精度 | 仅 DC 扫描，当前未使用 |
| `DBL_EPSILON` | `hisim2/hsm2evalenv.h:29` 等 | 2.22e-16 | 模型内部容差远低于 FP32 精度地板 | 仅 HISIM 模型，当前未使用 |
| `DBL_MAX` | `hfet1/hfettemp.c:96,101`, `mesa/` | 1.8e308 | 赋给 float → +Inf | HFET/MESA 模型，当前未使用 |
| `MIN_EXP` | 所有 BSIM3/4 家族 (20+ 份副本) | 1.71e-15 | `log(1.0f + 1.7e-15f) = log(1.0f) = 0`，亚阈值平滑函数崩溃 | **需要关注**：对低功耗/亚阈值偏置电路有影响。当前运放偏置在强反型，不受影响 |
| `MAX_EXP_ARG` | `defines.h:54`, bjt, mos1-9, soi3 | 709.0 | `expf(89) = +Inf`，但钳位值为 709 | 大信号瞬态才触发，DC 分析不受影响 |
| `N_MINLOG` | `noisedef.h:106` | 1e-38 | 低于 `FLT_MIN` (~1.17e-38) | 仅噪声分析 |
| `1e-20`, `7e-18` | bsim4/b4ld.c, bsimsoi 等 | 1e-20 | 远低于 FLT_MIN，冲刷为零 | 用作极小阈值保护，FP32 下需重校 |

### 3.4 isnan.c 中的 NaN 检测

| 属性 | 说明 |
|------|------|
| 位置 | `src/maths/misc/isnan.c:7-8` |
| 硬编码 | `union ieee754_double` 使用 64 位 double 的位字段布局 (11 指数位 + 52 尾数位) |
| FP32 下的问题 | 当 `SPICE_REAL = float` (32 位)，位字段布局完全不同 (8 指数位 + 23 尾数位) |
| 不改的理由 | 编译器内置的 `isnan()` (C99 math.h) 在 b4v5ld.c 中直接使用，绕过了这个文件。该文件仅在某些特定路径中被调用 |

### 3.5 cmath1.c 中的 `double complex`

| 属性 | 说明 |
|------|------|
| 位置 | `src/maths/cmaths/cmath1.c:881-882` |
| 硬编码 | `double complex midin` / `double complex midout` — C99 double complex 类型 |
| 不改的理由 | 仅用于复数反双曲正切，XSPICE 禁用后不会被调用 |

## 附：多模型同文件加载的结构体字段损坏问题

### 症状

FP32 构建下，当 p18 + n18 两个 `.model` 定义在**同一个 `.include` 文件**中时，第二个加载的模型（或其实例）的字段出现垃圾值：

```
here->BSIM4v5l = -1.0e33（应为 2.67e-6）
model->BSIM4v5xl = -3.36e15（应为 -1.3e-8）
here->BSIM4v5w = -1.07e-22（应为 6.07e-5）
```

分离 `.include` 文件（每个模型单独一个文件）可完全规避。

### 根因分析

**不是结构体对齐/填充问题。** 所有 struct 的 `sizeof()` 在编译时由编译器正确计算（FP32 下 SPICE_REAL=4 字节，FP64 下 8 字节）。`CKTmodCrt()` / `CKTcrtElt()` 使用 `sizeof(struct)` 分配，正确清零。

**最可能的根因**：Ngspice 输入预处理管线（`src/frontend/inpcom.c`）中预先存在的堆缓冲区溢出。

证据链：
1. 两个模型在同一文件中 → 它们的 card 结构体在堆上相邻
2. `inp_fix_param_values()`（~400 行）等函数在预处理阶段对 card 行缓冲区进行原地修改和重新分配
3. 某处缓冲区大小计算错误导致溢出，破坏了相邻 card 的数据
4. **FP64 下同样存在这个溢出**，但由于 BSIM4 结构体大一倍（~2000→~1000 字节），堆布局不同，溢出区域落在未使用或非关键字段上，未产生可见症状
5. FP32 下结构体更紧凑，溢出恰好覆盖到 `xl`、`l`、`w` 等关键字段

### 排查结论

| 检查项 | 结果 |
|--------|------|
| `struct sBSIM4v5instance` 布局 | ✅ `sizeof()` 由编译器计算，无硬编码偏移 |
| `struct sBSIM4v5model` 布局 | ✅ 同上 |
| `struct card` 中 `float w,l,nf` | ✅ 始终 4 字节，与 SPICE_REAL 无关 |
| `CKTmodCrt()` / `CKTcrtElt()` 分配 | ✅ 使用 `sizeof(struct)` + `calloc` 清零 |
| `INPmakeMod()` 指针存储 | ✅ `INPmodLine = line` 存储 card 指针 |
| `inp_stitch_continuation_lines()` | ✅ 拼接逻辑正确 |
| `inp_fix_param_values()` (~400行) | ⚠️ 最可能的溢出源——缓冲区大小计算复杂 |
| Hash table / DSTRING / tokenizer | ✅ 均使用精确分配 |

### 解决方案

| 方案 | 状态 | 说明 |
|------|------|------|
| A) **分离 .include 文件** | ✅ 已采用 | 每个模型单独 `.include`，完全规避 |
| B) 输入验证 + 钳位 | ✅ 已实施 | `b4v5set.c` 中 xl/xw 钳位 (`|x|>1e-3→0`) |
| C) ASan 定位修复溢出 | 🔲 后续 | `-fsanitize=address` 编译 + 测试可精确定位 |
| D) 替换 card 行指针为副本 | 🔲 后续 | `inpmkmod.c` 中 `strdup(line->line)` 代替指针 |

方案 A 是当前生产方案，已验证稳定。方案 C 可彻底修复，但需要 ASan 编译环境（在 WSL2 中可用）。

## 设计原则总结

```
┌──────────────────────────────────────────────────┐
│              FP64 保留决策树                       │
├──────────────────────────────────────────────────┤
│                                                    │
│  该模块是否被当前构建禁用？                         │
│    ├── 是 (KLU, XSPICE, CIDER) → 无需处理         │
│    └── 否 → 继续判断                               │
│                                                    │
│  该模块是否涉及浮点数值计算？                       │
│    ├── 否 (拓扑, 解析, 控制流) → 无需处理          │
│    └── 是 → 继续判断                               │
│                                                    │
│  该模块是否为第三方库且硬编码 double？              │
│    ├── 是 (AMD/BTF/COLAMD——仅符号算法) → 无需处理  │
│    └── 否 → 继续判断                               │
│                                                    │
│  该模块是否在当前场景下阻塞仿真？                   │
│    ├── 是 → 必须修复                                │
│    │   (已完成: Leff/Weff, xl/xw, DEVpnjlim, sqrt) │
│    └── 否 → 记录但不修改                            │
│        (裸数学函数, complex.h, FP64常量)            │
│                                                    │
└──────────────────────────────────────────────────┘
```

## 验证状态

针对 6 器件运放偏置电路（TSMC 0.18µm, p18/n18 BSIM4v5 模型）：

| 参数 | FP64 参考值 | FP32 结果 | 相对误差 |
|------|------------|-----------|----------|
| vb2 | 1.733067 | 1.733067 | 0% |
| vb3 | 1.732702 | 1.732703 | 0.00006% |
| vb1 | 0.03297214 | 0.03297193 | 0.0006% |
| Idd | -6.25717e-7 | -6.25696e-7 | 0.003% |

**结论**：在当前的 FP64 保留策略下，FP32 的模拟精度达到 0.003% 以内，满足仿真要求。
